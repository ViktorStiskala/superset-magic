The problem & the ask

The reverse-sync picker (pick_reverse_sync in src/tui/ui.rs) re-invokes inquire::Select::new(…).prompt() inside a loop. Every time you press Enter, inquire finalizes the prompt and prints a permanent "answered" line to scrollback — so each keypress stacks another Untracked files to push back to main: row. That is exactly the screenshot behavior.

Now — stacks on every Enter

— Reverse sync (untracked → main) — ✓ Untracked files to push back to main: ↳ show diff ✓ Untracked files to push back to main: [ ] .compound-engineering/config.local.yaml (differs) ? Untracked files to push back to main: [x] .compound-engineering/config.local.yaml (differs) ↳ show diff › ✓ Push selected

Proposed — one frame, redrawn in place

— Reverse sync (untracked → main) — [x] .compound-engineering/config.local.yaml (differs) [ ] apps/api/.dev.vars (new) [ ] .env.local (differs) ✓ Push selected (1) ␣ toggle · → show diff · ↑↓ move · ⏎ apply · Esc cancel

On top of fixing the re-print, this ideation covers two features the picker should grow:

Feature A — fixed picker + rich side-by-side diff

Redraw in place; Space toggles a row, → opens its diff. Side-by-side columns with per-side line numbers, ~3 lines of folded context around each change, ANSI color, headers “Local file” / “Main branch”, and intra-line highlighting.

Feature B — interactive Merge mode

Per file, cycle [push to main · pull from main · interactive merge · delete] with ←/→; a sensible default per file; navigate the side-by-side diff with ↑/↓. Mimics git mergetool: you decide what ends up in the local file, in main, or is deleted.

Grounding Context

Codebase Context

Where the current design stands

  • The re-print is structural, not a missing flag. inquire tracks its last_rendered_frame/current_frame inside one prompt() call; looping Select::prompt() leaves every submitted answer as scrollback. RenderConfig only reskins colors/prefixes — there is no "suppress the answer line" switch, and the prompt set is closed with no custom-key hook. verified: docs.rs/inquire + frame_renderer.rs
  • The current diff is git diff --no-index piped to a pager (git::diff_no_index_paged) — a unified diff, no side-by-side, no per-side line numbers, no custom titles.
  • Space-toggle was deliberately abandoned once. The team moved off inquire::MultiSelect to a Select-loop because action rows (+ Add new…, Done, show diff) don't work under MultiSelect's Space/Enter split (see docs/solutions/…/inquire-action-loop). The consequence: getting Space-toggle + →-diff + no-stacking all at once requires leaving inquire for a raw-terminal layer — that is the central design axis of this whole ideation.
  • The files are git-untracked secrets (.env, .dev.vars, magic.local.json). There is no common ancestor, so Merge is a 2-way reconcile, not a 3-way merge. reverse_sync::classify() already computes the byte-level split (DiffStatus::{Differs, WorktreeOnly, Identical}).
  • The terminal backend is already in the tree. crossterm 0.29 and console are already in Cargo.lock (via inquire's default features and supports-color). So a crossterm-based rewrite of just this one screen adds zero new terminal backend, and even ratatui adds essentially only itself. verified against real Cargo.lock

The verified crate landscape

Ten agents researched and adversarially verified the ecosystem against docs.rs, crates.io, and repo sources. The one-line shape of the answer: one diff engine dominates (similar), no crate renders a proper side-by-side terminal diff out of the box, and the interactive layer is a three-way choice (hand-rolled crossterm · ratatui framework · borrow a purpose-built component).

Past learnings that apply

  • The Select-loop action pattern (row-as-action) is the deliberate current convention; any raw-terminal rewrite should preserve its ergonomics (cursor stays on the row after a toggle) while dropping the per-Enter re-print.
  • Interactive TUI code is manual-smoke, not unit-tested here by convention — but the pure logic (diff computation, the merge assembly, the decision truth-table) can and should stay unit-testable behind seams, exactly as classify()/copy_candidate_into_main() already are.
  • Version bump is required on any user-visible change (CLAUDE.md) — implementing any idea below bumps the minor version so the self-updater ships it.

Topic Axes

The subject decomposes into five orthogonal surfaces. By the user's "both features per idea" constraint, every surviving idea spans axes A1–A4; the ideas genuinely diverge on A5 (footprint) and on how they realize A3 (rendering) and A4 (merge).

  • A1 · Re-print fix & in-place redraw — a single frame that repaints, never stacks.
  • A2 · Selection ergonomics — Space toggles, → opens the diff, arrows navigate.
  • A3 · Side-by-side rendering — two columns, per-side line numbers, ~3-line context fold, color, "Local file"/"Main branch" titles, intra-line highlight.
  • A4 · Interactive merge model — per-file decision cycle, base-less 2-way semantics, mtime as a soft default.
  • A5 · Dependency & architecture footprint — how much enters the tree and how large the blast radius is on the existing inquire-based design.

Shared foundation & what both features look like

All five architectures share the same spine, so it is described once here and not repeated per card:

  1. One diff engine — similar. TextDiff::from_lines(main, local).grouped_ops(3) yields context-folded hunks whose DiffOps carry both-side line indices — the same data drives the side-by-side view (Feature A) and the per-hunk merge decisions (Feature B). One diff, two consumers.
  2. inquire stays for every other one-shot prompt (menu, bootstrap, config edit). Only the reverse-sync/merge screen moves off it.
  3. The merge is 2-way, base-less (see the model below), with mtime as a suggestion, never authority.

Feature A concretely. Two columns with independent line numbers, unchanged runs folded to ~3 lines (similar::grouped_ops(3)), colored add/del rows, and word-level intra-line highlighting (iter_inline_changes). No crate ships this — it is a hand-rolled renderer over similar + console.

↑↓ move between files   ←→ choose action   →/Tab open side-by-side diff   Enter apply all   Esc cancel

Feature B concretely. Each file carries a segmented decision cycled with ←→; impossible actions are greyed for the file's state; the default is pre-selected from mtime (shown, never hidden). "Interactive merge" opens a hunk-level reconcile; "delete" asks an explicit scope.

The base-less 2-way merge model

Because these are untracked secrets with no shared history, the model is reconcile two states, not diff3. This is the semantic core every idea's Merge mode implements.

The merge model. With no ancestor, "merge" is ordered per-hunk selection over a 2-way diff (the technique git add -p and Meld use). Every write into main still funnels through the existing ensure_gitignored_in_main secret-safety step.

Verified constraint · the mtime default is a suggestion, not authority

The requested "default from mtime" is sound as a hint but unreliable as truth: std::fs::copy preserves mtime on macOS but not Linux, and git checkout/worktree add reset mtime to the checkout instant. So: always content-hash first (classify() already does — identical files never reach the picker), show both timestamps, and on a tie or untrustworthy signal default to Interactive merge, not a guessed direction.

The design space — where the five ideas sit

Reading the map. Left = cheapest footprint (crossterm is already present); up = the merge is resolved inside the terminal rather than by handing off to an editor. ★ Idea 1 is the top pick: it lands highest on both the richness the ask implies and the strongest prior art.

RANKED #1 · TOP PICK

Idea 1 — ratatui “merge cockpit”

Axis: terminal layer = full TUI Confidence 85% Complexity High

Replace the reverse-sync screen with a single full-screen ratatui app. A left file-list pane shows each candidate with its decision chip; a right diff pane renders the live side-by-side view for the highlighted file. Both features live in one cockpit — the diff view is the merge view is the picker — which is exactly the "side-by-side diff is displayed but navigation is allowed" experience the ask describes.

+ ratatui 0.30+ similar 3.1.1 crossterm 0.29 (present)console (present) prior art: gitui
Feature A

List+ListState give the in-place, arrow-navigable picker (Space toggles state held in an App struct — no re-print by construction). Layout::horizontal([50,50]) makes the two titled columns; Paragraph + scroll renders grouped_ops(3) hunks; Span styling carries color + iter_inline_changes highlights.

Feature B

A per-file Decision enum cycled with ←/→; default from mtime. "Interactive merge" opens an in-pane hunk walk (keep local / keep main / keep both) assembled from the same DiffOp stream. ↑/↓ moves files/hunks; "delete" prompts an explicit scope.

external: gitui (22k★) ships this exact "file list + hunk-level diff + per-hunk action" pattern on ratatui 0.30 + crossterm 0.29 — verified from its Cargo.toml — so the shape is proven, not speculative. reasoned: crossterm is already resolved to 0.29 in Cargo.lock, so ratatui adds one crate rather than a new backend.

Rationale. This is the only option that unifies both features into one navigable screen instead of a picker plus a separate diff pager plus a separate merge step. Immediate-mode redraw makes the re-print impossible and makes "repaint on every keypress" (toggle, arrow, decision change) the natural model rather than something you fight.

Pros
  • Fullest match to the ask: live navigable side-by-side + in-place picker + in-TUI merge, one screen.
  • Verified-cheap: one new framework crate; backend already present.
  • Strong, current prior art (gitui) for the exact interaction shape.
  • Immediate-mode redraw is a natural fit for a keypress-driven screen.
Cons
  • Most code & largest new surface for a small CLI — the richest but heaviest option.
  • ratatui's widget/text/style modules + a one-time immediate-mode learning curve.
  • The in-TUI hunk picker is the most bespoke part and needs its own careful manual-smoke pass.
  • Biggest binary-size delta of the five.
Effort & sequencing notes

Ship in two commits: (1) the picker+diff pane (Feature A) — already a complete, valuable fix for the re-print and the diff view; (2) the Merge decision + in-TUI hunk assembly (Feature B). Keep the merge assembly (walk grouped_ops, concatenate chosen slices) as a pure, unit-tested function behind the TUI, mirroring how classify()/copy_candidate_into_main() are already testable. Route all four decisions through the existing ensure_gitignored_in_main seam.

Confidence: 85%  ·  Complexity: High  ·  New crates: ratatui, similar

RANKED #2

Idea 2 — crossterm surgical picker + $EDITOR merge handoff

Axis: merge = editor handoff · min footprint Confidence 82% Complexity Medium

The smallest-blast-radius option. Replace only pick_reverse_sync with a hand-rolled crossterm raw-mode / alternate-screen loop — since crossterm is already in the tree, this adds no backend. The list redraws in place (re-print gone); Space toggles; → opens a full-screen side-by-side diff overlay and Esc returns. For Merge, ←/→ cycles the per-file decision; "interactive merge" synthesizes a conflict-marked temp file and hands off to the user's $EDITOR / git mergetool.

+ similar 3.1.1 crossterm 0.29 (present)console (present) tempfile (present)prior art: git add -p “e”
Feature A

Own redraw loop (move cursor home, clear, reprint) — no stacking. Space/→ dispatched from raw KeyEvents. The → overlay renders the two-column diff from similar::grouped_ops(3), with console::pad_str for Unicode-correct columns and color.

Feature B

←/→ cycles [push · pull · merge · delete] inline (default from mtime). "Merge" writes <<< local / === / >>> main markers to a tempfile, launches $EDITOR, re-reads, and rejects the result if any marker line remains.

external: git add -p's e (edit-hunk) escape hatch is the canonical "when the picker can't decide, hand it to the editor" pattern. reasoned: the hard, freeform part of merging is exactly what an editor does best; offloading it removes the riskiest code while keeping push/pull/delete fast and in-terminal.

Rationale. Matches the existing architecture's grain — one function changes, inquire stays for everything else, and the "show diff via a shell-out/pager" convention is preserved (just upgraded to side-by-side). Lowest risk to ship.

Pros
  • Tiny footprint: one new crate (similar); no new backend; blast radius is a single function.
  • Offloads the riskiest work (freeform reconcile) to tools users already know.
  • Preserves the current Select-loop mental model, minus the re-print.
  • Fastest to implement and review.
Cons
  • The merge step leaves the TUI — a context switch, and it depends on $EDITOR/git mergetool being configured.
  • You hand-roll the overlay's scroll/layout (crossterm has no widgets).
  • Scales badly if you later want a persistent split-pane cockpit — that's re-deriving ratatui by hand.
  • Two contexts for the user (in-TUI vs editor) is less cohesive than Idea 1.

Confidence: 82%  ·  Complexity: Medium  ·  New crates: similar

RANKED #3

Idea 3 — crossterm self-contained in-TUI merge (no editor, no framework)

Axis: merge = self-contained in-TUI Confidence 70% Complexity Medium-High

Idea 2's minimal footprint, but the merge never leaves the terminal. On crossterm (no framework), build a small hunk-walk overlay: for each differing hunk the user presses a key to keep local / keep main / keep both, and the result is assembled from the similar DiffOp stream — no $EDITOR dependency at all. The right answer when you can't assume a developer's editor is configured (a self-updating CLI often can't).

+ similar 3.1.1 crossterm 0.29 (present)console (present) prior art: git add -p loop
Feature A

Identical to Idea 2: own redraw loop, Space toggle, → side-by-side overlay from grouped_ops(3), colored columns via console.

Feature B

"Interactive merge" enters a hunk-by-hunk overlay driven entirely by keys — j/k to move hunks, ←/→ to pick the side, live preview of the assembled file. ~50–100 LOC of "walk hunks, concatenate chosen slices" glue over similar. No editor, no external tool.

reasoned: similar::grouped_ops hands you exactly the per-hunk decision points; assembling a merged file from per-hunk choices is a short, pure, unit-testable function (the same shape git add -p and Meld are — bespoke glue over a diff engine). external: Meld's two-way mode proves per-chunk directional selection with no base is a coherent, shippable UX.

Rationale. Keeps the dependency story as small as Idea 2 and stays fully in-terminal and self-contained, which matters for a tool that can't assume $EDITOR. The cost is that you own the hunk-picker's redraw/scroll on raw crossterm.

Pros
  • Smallest deps and fully self-contained — no $EDITOR/git mergetool reliance.
  • Cohesive single-tool experience; no context switch mid-merge.
  • Merge-assembly logic is pure and unit-testable behind the TUI.
Cons
  • Most bespoke of the crossterm options — you hand-build the hunk overlay's scroll, cursor, and preview.
  • No gitui-scale prior art for a hand-rolled crossterm hunk picker (unlike ratatui's).
  • Re-implements a slice of what ratatui/scm-record already provide; if scope grows, Idea 1 becomes the better home.
  • Freeform "type something neither side has" edits aren't possible without an editor fallback.

Confidence: 70%  ·  Complexity: Medium-High  ·  New crates: similar

RANKED #4

Idea 4 — embed scm-record (borrow the hunk-selection UI)

Axis: terminal layer = borrowed component Confidence 62% Complexity Medium · spike-first

Don't build the hardest part — adopt it. scm-record is a maintained, ratatui-based, embeddable widget purpose-built for "interactively select which hunks/lines to keep between two versions of a file with no common ancestor" — the exact base-less 2-way problem ss-magic has. It already ships as a production git difftool (scm-diff-editor) inside Jujutsu and git-branchless (git record -i).

+ scm-record 0.10(+ similar, optional) crossterm 0.29 (present)used by: Jujutsu · git-branchless
Feature A

Renders as a ratatui/crossterm component that redraws in place (re-print solved for free). Its default is a hunk list; a genuine two-column side-by-side layout is unconfirmed and may need patching or a contribution upstream.

Feature B

Recorder/RecordState with Section hunks and a Tristate (full/none/partial) selection — very close to per-hunk keep-local/keep-main. Needs a thin adaptation layer to map its stage-vs-not model onto ss-magic's 3-outcome [push · pull · merge].

external: verified from docs.rs — scm-record depends on ratatui 0.30 + crossterm 0.29 and exposes Tristate/Section/SelectedContents; it is the difftool front-end for Jujutsu. reasoned: the merge-UI is the single most bespoke, highest-risk piece of every other idea — borrowing a proven implementation removes that risk if the fit holds.

Rationale. Potentially the least merge-UI code of any option. But it is niche (105★), its native semantics are binary stage/don't-stage rather than a 3-way outcome, and its side-by-side support is unverified — so this earns a hands-on spike (embed against a synthetic two-file case) before committing, not a blind adoption.

Pros
  • Purpose-built for base-less per-hunk selection — the exact problem, already solved.
  • Production-proven inside Jujutsu & git-branchless.
  • Backend already present; adds the component, not a new terminal stack.
Cons
  • Niche (105★, thin docs) — expect to read source and possibly pin/patch it.
  • Native model is stage/don't-stage, not [push·pull·merge] — needs an adaptation layer.
  • Side-by-side layout & custom titles unconfirmed — may require an upstream contribution.
  • Cedes control of the UX to an external component's evolution.

Confidence: 62%  ·  Complexity: Medium (spike-first)  ·  New crates: scm-record

RANKED #5

Idea 5 — diffy editor-native conflict merge

Axis: merge = editor-native markers Confidence 65% Complexity Medium

A variant of the editor-handoff path (Idea 2) that leans on diffy to make the merge feel exactly like resolving a git conflict in your editor. The picker + side-by-side preview run on crossterm + similar; "interactive merge" produces git-native <<</===/>>> conflict markers that VS Code, vim, etc. recognize with their built-in accept-current / accept-incoming / accept-both affordances — near-zero merge-UI code.

+ similar 3.1.1+ diffy 0.5 crossterm 0.29 (present)tempfile (present)
Feature A

Same crossterm picker + similar side-by-side preview as Idea 2.

Feature B

Uses diffy's conflict-style formatting (via an empty/synthetic ancestor) to emit editor-native markers, so the "interactive merge" file opens with the editor's own conflict UI. Push/pull/delete stay in-TUI.

external: git-native conflict markers are the most widely understood merge UX; every mainstream editor renders accept-current/incoming buttons on them. reasoned (with a verified caveat): diffy::merge is a 3-way API, but the verifier confirmed an empty ancestor is mechanically valid and yields markers for a 2-way case — a legitimate trick, though undocumented for 2-way, so it must be treated as leaning on unspecified behavior.

Rationale. Best possible merge ergonomics for the least merge-UI code — but it adds a second diff engine (diffy alongside similar) and depends on diffy's empty-ancestor behavior, which keeps its confidence below the self-contained options. Markers can equally be hand-rolled from similar hunks if you'd rather not add diffy.

Pros
  • Merge output is instantly familiar — the editor's native conflict UI does the work.
  • Almost no bespoke merge code; diffy is mature (2026 release).
  • Same low-risk crossterm picker as Idea 2 for Feature A.
Cons
  • Two diff engines in the tree (similar + diffy) for overlapping purposes.
  • Relies on diffy's empty-ancestor behavior — mechanically valid but not documented for 2-way.
  • Still leaves the TUI for the merge; still needs a configured editor.
  • Marker-in-a-secret hazard: must reject any saved file that still contains a marker line.

Confidence: 65%  ·  Complexity: Medium  ·  New crates: similar, diffy
